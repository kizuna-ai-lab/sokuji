# P5: DML Enablement in ORT TTS Backends + cuDNN Officialization Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make device `"dml"` flow through all three heavy ORT TTS backends (MOSS-TTS-Nano, Supertonic 3, Qwen3-TTS) so the Windows DirectML sidecar SKU accelerates TTS on non-NVIDIA GPUs (spec D1, D2), and replace the hand-written cuDNN preload shim plus the Electron `LD_LIBRARY_PATH` injection with ORT's official `onnxruntime-gpu[cuda,cudnn]` extras + `onnxruntime.preload_dlls()` (spec D8).

**Architecture:** New `gpu-dml` `Deployment` rows (Windows-only via P3's `platforms` tag) sit in their own tier ranked just below `gpu-cuda` (`TIER_RANK`: `gpu-dml` 2.5 vs `gpu-cuda` 3.0 — both already defined); `TIER_DEVICE["gpu-dml"] == "dml"` hands each backend `device="dml"`, which each backend maps to the ORT provider list `["DmlExecutionProvider", "CPUExecutionProvider"]`. ALL graphs run on DML, including the autoregressive ones (D2 — no pre-emptive AR→CPU routing); Qwen3-TTS keeps its existing cold-graph (speaker encoder / tokenizer-12hz encode / text project) CPU split unchanged, mirroring the CUDA path. For MOSS the DML provider-list logic lives in the vendored `moss_tts/ort_runtime.py` (which the backend feeds a device *label*); Supertonic and Qwen3-TTS build the list themselves. cuDNN: `setup.sh` installs `onnxruntime-gpu[cuda,cudnn]==1.23.2` and `__main__` calls `onnxruntime.preload_dlls()` at startup when the CUDA EP is present; `_cudnn_preload.py` and the Electron `withTorchCudaLibs`/`nvidiaLibDirs` env surgery are deleted.

**Tech Stack:** onnxruntime-gpu 1.23.2 (NVIDIA SKU, Linux/Windows — `preload_dlls(cuda, cudnn, msvc, directory)` verified present in the dev venv), onnxruntime-directml 1.24.4 (Windows non-NVIDIA bundle, installed by P7 packaging, never on the Linux dev box), pytest with stubbed `onnxruntime` modules / fake sessions for every DML assertion, vitest for the Electron side.

**Outcome mapping (from the workstream brief):** Task 1 = outcome 1 (MOSS: device dml + ort_runtime dml enum + dml-session verification); Task 2/3 = outcome 1 (Supertonic, Qwen3 dml + cold/hot split); Task 4 = outcomes 2 & 3 (catalog gpu-dml rows Windows-only ranked below cuda; accel surfaces the tier and the "cuda == any accelerator" override pins dml); Tasks 5-6 = outcome 4 (cuDNN officialization, sidecar + Electron). Outcome 5 (stub-tested on Linux; deferred Windows verification) is satisfied by every DML assertion being a provider-list/enum stub plus the final deferred-verification section.

```python
def _resolve_ort_providers(execution_provider: str) -> list[Any]:
    normalized = _normalize_execution_provider(execution_provider)
    if normalized == EXECUTION_PROVIDER_CPU:
        return ["CPUExecutionProvider"]
    available_providers = set(ort.get_available_providers())
    if normalized == EXECUTION_PROVIDER_DML:
        # DirectML SKU (spec D2): ALL graphs on DML, including the autoregressive
        # ones. No preload_dlls (that is CUDA/cuDNN-only).
        if "DmlExecutionProvider" not in available_providers:

            raise RuntimeError(
                "DmlExecutionProvider was requested, but this onnxruntime build does not expose it. "
                "Install onnxruntime-directml that matches this platform. "
                f"Available providers: {available}"

    if "CUDAExecutionProvider" not in available_providers:
        available = ", ".join(ort.get_available_providers()) or "none"

            "Install onnxruntime-gpu that matches your CUDA/cuDNN runtime. "
            f"Available providers: {available}"
        )
```

```python
def default_providers(device: str | None = None) -> list[str]:
    """Resolve the ONNX Runtime execution providers for `device`.

    "dml" pins DirectML (Windows non-NVIDIA SKU) for the HOT graphs;
    build_sessions keeps the cheap one-shot COLD graphs on CPU regardless
    (spec D2: the autoregressive HOT graphs run on DML, the COLD ones stay
    CPU). A "dml" device never appends CUDA even when the CUDA EP is present."""
    import onnxruntime as ort

    available = ort.get_available_providers()
    dev = str(device).lower() if device else ""
    if dev == "cpu":
        return ["CPUExecutionProvider"]
    providers: list[str] = []
    if dev == "dml":
        if "DmlExecutionProvider" in available:
            providers.append("DmlExecutionProvider")
    elif "CUDAExecutionProvider" in available:
        providers.append("CUDAExecutionProvider")
    providers.append("CPUExecutionProvider")
    return providers
```
